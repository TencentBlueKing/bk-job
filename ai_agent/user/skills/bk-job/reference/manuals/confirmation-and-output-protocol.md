# 操作确认协议与输出规范

本手册约定：通过 [`scripts/job_apigw_client.py`](../../scripts/job_apigw_client.py) 的 **`plan-execute`** 调用网关 **`POST /api/v3/execute_job_plan`** 时，智能体须遵守的**确认门禁**与**对用户呈现方式**。

---

## 1. 操作确认协议

### 1.1 核心原则

**所有写操作（`plan-execute` / `execute_job_plan`）必须在执行前获得用户的明确确认。读操作（`cron-search`、`plan-search`、`plan-detail`、`cron-last-run`、`instance-status` 等）无需确认，直接执行。**

### 1.2 操作分级

| 风险等级 | 操作类型 | 确认方式 |
|---------|---------|---------|
| 🟢 低风险 | 搜索方案、搜索定时任务、查看详情、查看状态、查看日志 | 无需确认，直接执行 |
| 🟡 中风险 | **启动执行方案**（`plan-execute`） | 展示操作摘要 → 用户确认 → 执行 |

### 1.3 确认门禁（GATE CHECK）

> **🚫 硬性规则：在调用 `plan-execute`（真实执行，非 `--dry-run`）之前，必须通过以下门禁检查，缺一不可。违反任何一条都禁止调用。**

| # | 检查项 | 说明 |
|---|--------|------|
| G1 | 已向用户展示过确认摘要（格式见 **1.4**） | 当前回复或历史回复中必须包含完整的确认摘要 |
| G2 | 用户在**看到确认摘要之后**明确回复了确认意图 | 确认必须发生在摘要展示之后，不能是之前的回复 |
| G3 | 确认摘要与即将调用的参数一致 | 方案 ID、资源范围（`bk_scope_type` / `bk_scope_id`）、变量值必须与摘要中展示的一致 |
| G4 | 全局变量使用 `name` 标识且来源正确 | 变量须用 `name`（非仅 `id`）传递，且取自**当前方案**最新一次 `plan-detail`（`get_job_plan_detail`）的返回结果 |

**关于 G2 的特别说明（硬性规则）**：

**🚫 无论用户的启动意图表达多么明确、多么强烈，只要操作分级（**1.2**）要求确认，就必须：**

1. **先展示确认摘要**（**1.4** 格式）
2. **等待用户在看到摘要之后的独立回复**作为确认

以下表达方式**全部只算启动意图，不算确认**，即使措辞中包含「立即」「马上」「确认」等词：

- 「启动吧」/「执行吧」/「跑一下」
- 「更正后再启动一次」/「用新参数启动」
- 「立即启动」/「马上执行」/「直接跑」
- 「确认启动」/「确认执行」（如果此前**没有**展示过**当前参数**的确认摘要）

**正确处理流程**：

```
用户：任何形式的启动意图（包括「立即启动」「更正后再启动」等）
  ↓
AI 动作：展示确认摘要（1.4 格式），结尾加「请确认是否立即启动。」
  ↓
⛔ STOP — 必须等待用户的下一条独立回复
  ↓
用户（新的一条消息）：「确认」/「好的」/「没问题」/「启动」
  ↓
AI 动作：通过 G2 门禁，继续检查 G1/G3/G4，全部通过后调用 plan-execute（可先 --dry-run 核对再执行）
```

**反面案例（严禁）**：

```
用户：「更正后再启动一次」
  ↓
❌ AI 在同一轮回复中既展示摘要又执行启动 — 违反 G2
❌ AI 将用户的启动意图当作确认直接执行 — 违反 G2
```

### 1.4 确认摘要格式

在调用 **`plan-execute`（非 dry-run）** 之前，必须展示以下格式的摘要：

```
即将执行的操作：
- 操作类型：启动执行方案
- 方案名称：{方案名称}（ID: {job_plan_id}）
- 资源范围：{bk_scope_type} / {bk_scope_id}
- 全局变量：
  | 变量名 | 值 | 来源 |
  |--------|-----|------|
  | {name} | {value} | [方案默认] / [用户提供] / [业务记忆预填] |
- 步骤概览：共 {n} 个步骤

请确认是否立即启动。
```

敏感值可脱敏；主机类变量可摘要为「共 N 台」或列举 IP 片段。

### 1.5 确认结果处理

| 用户反馈 | 处理方式 |
|---------|---------|
| 确认执行（「确认」「好的」「没问题」等，且已在**上一轮或更早的回复中**看到过**当前参数**的确认摘要） | 通过 G2 门禁，执行 `plan-execute`，并反馈执行结果（含 `job_instance_id` 等） |
| 启动意图——**无论措辞多么明确**（「启动吧」「立即启动」「更正后再启动」等），只要**当前参数**的确认摘要**尚未展示过** | 展示确认摘要，**⛔ STOP 等待用户下一条独立回复确认** |
| 请求修改 | 按用户要求调整后，重新展示摘要供再次确认 |
| 取消操作 | 不执行，明确告知用户已取消 |

> **⚠️ 关键区分**：判断用户回复是「确认」还是「启动意图」，唯一标准是**当前参数的确认摘要是否已在之前的回复中展示过**。与用户措辞的强烈程度无关。

**与 `--dry-run` 的关系**：`plan-execute --dry-run` 为只读请求体校验，**不视为写操作完成**；真实启动仍须满足**第 1 节**全部门禁。

## 2. 输出规范

### 2.1 禁止暴露内部推理过程

脚本执行、网关调用、本地 `jq` 过滤等是**后台行为**，对用户不可见。**禁止**在输出中叙述这些过程。

| 禁止 | 正确做法 |
|------|---------|
| 「让我先搜索一下执行方案……」 | 直接展示搜索结果 |
| 「正在调用 API 查询状态……」 | 直接展示状态 |
| 「让我获取一下方案详情……」 | 直接展示方案详情 |

### 2.2 结构化交付

每次输出应该是**给用户看的结构化成果**，而非思考日志。

| 规则 | 说明 |
|------|------|
| **结果导向** | 输出表格、状态、日志摘要，而非过程描述 |
| **格式统一** | 表格、列表为主，避免大段散文 |
| **层次分明** | 标题分级，便于定位 |

### 2.3 交互式工作流

| 规则 | 说明 |
|------|------|
| **逐步推进** | 搜索 → 详情/参数 → **确认摘要** → 启动，按序推进；关键节点等待用户 |
| **启动路径禁止抢跑** | **不得**在同一轮回复内完成「展示确认摘要」与「真实 plan-execute」（违反 G2）；`--dry-run` 可与摘要同轮用于展示请求体，但**非**最终执行 |
| **禁止替用户决策** | 多方案/多定时任务歧义、参数空缺时，须等用户选定或补充 |

> 与 [listing-and-token-efficient-analysis.md](listing-and-token-efficient-analysis.md) 的关系：列举类查询仍应**分页/本地过滤**以省 Token；**2.3** 约束的是**带确认的启动链路**，不要求把整页原始 JSON 贴进对话。

### 2.4 展示规范

| 内容 | 格式要求 |
|------|----------|
| **执行方案搜索结果** | 表格：ID、名称、创建人、最后修改时间（有则填）；必要时加 `need_update` |
| **定时任务搜索结果** | 表格：ID、名称、启停状态、Cron（有则填）、创建人或最后修改（有则填） |
| **方案详情** | 分「步骤列表」与「全局变量」两部分 |
| **执行状态** | 清晰语义（如「执行成功」「执行中」「执行失败」），附各步骤状态 |
| **执行日志** | 按步骤与主机（或容器）分组展示 |
