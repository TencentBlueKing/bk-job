# 搜索执行方案并启动

## 概述

本 Skill 引导 AI 帮助普通用户通过自然语言完成"搜索执行方案 → 确认详情 → 启动执行 → 查看结果"的完整工作流。所有实际 API 调用通过 Job MCP Server 的 tools 完成。

| 能力 | 说明 |
|------|------|
| **方案搜索** | 按关键词模糊搜索执行方案列表 |
| **详情查看** | 查看执行方案的步骤列表和全局变量定义 |
| **参数溯源** | 每个变量标注来源：[方案默认] / [用户提供] / [业务记忆预填] / [待确认] |
| **操作确认** | 遵循操作确认协议，启动前必须用户确认 |
| **状态跟踪** | 查看作业实例执行状态和日志 |
| **业务记忆** | 按业务维度沉淀常用方案和参数偏好 |

## §0 环境就绪检查

**每次 Skill 启动时，在执行任何工作流程（§6）之前，必须按顺序完成以下全部检查。任何一步未通过都必须先解决，不得跳过。**

### 检查流程

1. **检查 MCP 服务可用性**：确认本 Skill 所需的 3 个 Job MCP Server 均已配置
   - 尝试列出可用的 MCP Server 列表，或对所需 Server 发起一次轻量调用（如 `get_job_plan_list` 传空关键词）
   - ✅ 全部就绪 → 继续
   - ❌ 存在缺失 → **阻塞后续流程**，按 §1.5 引导用户配置

2. **检查业务信息**：确认用户是否提供了业务 ID
   - ✅ 已提供 → 继续
   - ❌ 未提供 → 询问用户业务 ID

3. **加载业务记忆**（§5.3 强制检查点）：尝试读取业务记忆文件
   - 使用 `read_file` 工具读取 `{SKILL_DIR}/memory/businesses/{业务ID}.md`
   - 文件存在 → 加载内容用于后续预填
   - 文件不存在 → 标记为初识期，流程结束后触发 §5.5 记忆沉淀引导
   - **此步骤不阻塞后续流程**，但必须执行

### 缺失 MCP 服务时的标准话术

## §1 MCP 服务与工具清单

### 1.1 MCP 服务概览

本 Skill 依赖 3 个 Job MCP Server：

| MCP Server | 名称 | 定位 |
|---|---|---|
| `jobv3-cloud-prod-mcp-resources` | 蓝鲸作业平台MCP-资源类操作 | 执行方案查询、定时任务查询 |
| `jobv3-cloud-prod-mcp-start` | 蓝鲸作业平台MCP-启动类操作 | 启动执行方案 |
| `jobv3-cloud-prod-mcp` | 蓝鲸作业平台MCP-任务查询类操作 | 任务状态查询、执行日志查看 |

| Tool | 功能 | 关键参数 |

| `get_cron_list` | 查询业务下定时任务信息 | `bk_scope_type`、`bk_scope_id` |

| Tool | 功能 | 关键参数 |
|------|------|----------|
| `execute_job_plan` | 启动执行方案 | `job_plan_id`、`bk_scope_type`、`bk_scope_id`、`global_var_list`（可选） |

**任务查询域 `jobv3-cloud-prod-mcp`**：

| Tool | 功能 | 关键参数 |
|------|------|----------|

| 查看日志 | | | ● |

### 1.4 调用规则

1. **使用 MCP 工具**：根据操作类型调用对应 MCP Server
2. **ID 传递规则**：所有 ID 类型参数必须传递原始数字，禁止使用科学计数法
3. **🚫 全局变量传递规则（硬性规则）**：调用 `execute_job_plan` 时，`global_var_list` 中的每个变量**必须使用 `name` 字段标识变量，禁止使用 `id` 字段**。原因：不同执行方案的变量 `id` 不同，跨方案复用 `id` 会导致参数错乱。`name` 是稳定且语义明确的标识。
   - ✅ 正确：`{"name": "hosts", "server": {...}}`
   - ❌ 错误：`{"id": 1066428, "server": {...}}`
4. **变量值必须取自当前方案**：每次启动前必须使用**最新一次 `get_job_plan_detail` 返回的变量信息**构建参数，严禁复用之前其他方案的变量 ID 或缓存数据

## §2 核心概念

### 资源范围（必须先确认）

每次操作都需要指定资源范围，用户必须提供以下信息：

| 参数 | 说明 | 示例 |
|------|------|------|
| `bk_scope_type` | 资源范围类型 | `biz`（业务）或 `biz_set`（业务集） |
| `bk_scope_id` | 资源范围 ID | 业务 ID 如 `"2"`，业务集 ID 如 `"9991001"` |

> **提示**：如果用户没有主动提供，必须先询问"您要操作哪个业务？请提供业务 ID"。大多数用户使用的是 `biz` 类型。若业务记忆中已有该业务的常用方案信息，可利用记忆辅助预填。

### 执行方案（Job Plan）

执行方案是作业模板的一个可执行实例，包含：
- **步骤列表**：脚本执行、文件分发等步骤的有序组合
- **全局变量**：执行时可覆盖的参数，包括字符串变量、主机变量等

### 全局变量类型

| 类型 | 说明 | 传参方式 |
|------|------|----------|
| 字符串 | 普通文本变量 | `{"name": "变量名", "value": "值"}` |
| 密码 | 敏感信息 | `{"name": "变量名", "value": "值"}` |
| 数组 | 空格分隔的数组 | `{"name": "变量名", "value": "val1 val2 val3"}` |
| 命名空间 | 命名空间变量 | `{"name": "变量名", "value": "值"}` |
| 主机 | 执行目标主机 | `{"name": "变量名", "server": {"ip_list": [{"bk_cloud_id": 0, "ip": "127.0.0.1"}]}}` |

## §3 操作确认协议

### 3.1 核心原则

**所有写操作（`execute_job_plan`）必须在执行前获得用户的明确确认。读操作（查询、获取信息）无需确认，直接执行。**

### 3.2 操作分级

| 风险等级 | 操作类型 | 确认方式 |
|---------|---------|---------| 
| 🟢 低风险 | 搜索方案、查看详情、查看状态、查看日志 | 无需确认，直接执行 |
| 🟡 中风险 | 启动执行方案 | 展示操作摘要 → 用户确认 → 执行 |

### 3.3 确认门禁（GATE CHECK）

> **🚫 硬性规则：在调用 `execute_job_plan` 之前，必须通过以下门禁检查，缺一不可。违反任何一条都禁止调用。**

| # | 检查项 | 说明 |
|---|--------|------|
| G1 | 已向用户展示过确认摘要（格式见 §3.4） | 当前回复或历史回复中必须包含完整的确认摘要 |
| G2 | 用户在**看到确认摘要之后**明确回复了确认意图 | 确认必须发生在摘要展示之后，不能是之前的回复 |
| G3 | 确认摘要与即将调用的参数一致 | 方案 ID、业务 ID、变量值必须与摘要中展示的一致 |
| G4 | 全局变量使用 `name` 标识且来源正确 | 变量必须用 `name`（非 `id`）传递，且取自**当前方案**最新一次 `get_job_plan_detail` 的返回结果 |

**关于 G2 的特别说明（硬性规则）**：

**🚫 无论用户的启动意图表达多么明确、多么强烈，只要操作分级（§3.2）要求确认，就必须：**
1. **先展示确认摘要**（§3.4 格式）
2. **等待用户在看到摘要之后的独立回复**作为确认

以下表达方式**全部只算启动意图，不算确认**，即使措辞中包含"立即""马上""确认"等词：
- "启动吧" / "执行吧" / "跑一下"
- "更正后再启动一次" / "用新参数启动"
- "立即启动" / "马上执行" / "直接跑"
- "确认启动" / "确认执行"（如果此前没有展示过当前参数的确认摘要）

**正确处理流程**：

```
用户：任何形式的启动意图（包括"立即启动""更正后再启动"等）
  ↓
AI 动作：展示确认摘要（§3.4 格式），结尾加"请确认是否立即启动。"
  ↓
⛔ STOP — 必须等待用户的下一条独立回复
  ↓
用户（新的一条消息）："确认" / "好的" / "没问题" / "启动"
  ↓
AI 动作：通过 G2 门禁，继续检查 G1/G3/G4，全部通过后调用 execute_job_plan
```

**反面案例（严禁）**：
```
用户："更正后再启动一次"
  ↓
❌ AI 在同一轮回复中既展示摘要又执行启动 — 违反 G2
❌ AI 将用户的启动意图当作确认直接执行 — 违反 G2
```

### 3.4 确认摘要格式

在执行 `execute_job_plan` 之前，必须展示以下格式的摘要：

```
即将执行的操作：
- 操作类型：启动执行方案
- 方案名称：{方案名称}（ID: {job_plan_id}）
- 业务：{bk_scope_id}
- 全局变量：
  | 变量名 | 值 | 来源 |
  |--------|-----|------|
  | {name} | {value} | [方案默认] / [用户提供] / [业务记忆预填] |
- 步骤概览：共 {n} 个步骤

请确认是否立即启动。
```

### 3.5 确认结果处理

| 用户反馈 | 处理方式 |
|---------|---------| 
| 确认执行（"确认"、"好的"、"没问题"等，且已在**上一轮或更早的回复中**看到过当前参数的确认摘要） | 通过 G2 门禁，执行操作，并反馈执行结果 |
| 启动意图——**无论措辞多么明确**（"启动吧"、"执行吧"、"立即启动"、"更正后再启动"、"重新启动一次"等），只要当前参数的确认摘要**尚未展示过** | 展示确认摘要，**⛔ STOP 等待用户下一条独立回复确认** |
| 请求修改 | 按用户要求调整后，重新展示摘要供再次确认 |
| 取消操作 | 不执行，明确告知用户已取消 |

> **⚠️ 关键区分**：判断用户回复是"确认"还是"启动意图"，唯一标准是**当前参数的确认摘要是否已在之前的回复中展示过**。与用户措辞的强烈程度无关。

## §4 输出规范

### 4.1 禁止暴露内部推理过程

工具调用、数据搜索等过程是**后台行为**，对用户不可见。禁止在输出中叙述这些过程。

| 禁止 | 正确做法 |
|------|---------| 
| "让我先搜索一下执行方案……" | 直接展示搜索结果 |
| "正在调用 API 查询状态……" | 直接展示状态 |
| "让我获取一下方案详情……" | 直接展示方案详情 |

### 4.2 结构化交付

每次输出应该是**给用户看的结构化成果**，而非 AI 的思考日志。

| 规则 | 说明 |
|------|------|
| **结果导向** | 输出的是搜索结果表格、方案详情、执行状态，不是过程描述 |
| **格式统一** | 使用表格、列表等结构化格式，避免大段散文 |
| **层次分明** | 用标题分级组织内容，便于用户快速定位 |

### 4.3 交互式工作流

| 规则 | 说明 |
|------|------|
| **逐步推进** | 按工作流程的 Step 顺序执行，每步完成后等待用户指示 |
| **禁止一次性输出** | 不允许把搜索、详情、启动结果在一次回复中全部输出 |
| **禁止替用户决策** | 需要用户选择的环节（如选方案、确认参数），必须等用户回复 |

### 4.4 展示规范

| 内容 | 格式要求 |
|------|---------|
| **搜索结果** | 表格形式，包含 ID、名称、创建人、最后修改时间 |
| **方案详情** | 分"步骤列表"和"全局变量"两部分展示 |
| **执行状态** | 使用清晰的状态描述（如"执行成功"、"执行中"、"执行失败"），附各步骤状态 |
| **执行日志** | 按步骤和主机分组展示 |

## §5 业务记忆协议

### 5.1 记忆存储基准路径

记忆文件存储在**本 Skill 所在目录**下的 `memory/` 子目录中。

> **基准路径**：`{SKILL_DIR}/memory/`
> 其中 `{SKILL_DIR}` 是本 SKILL.md 文件所在的目录。

```
{SKILL_DIR}/
├── SKILL.md              # 本文件
└── memory/               # 记忆存储根目录
    └── businesses/        # 业务维度知识
        └── {业务ID}.md    # 各业务的专属知识
```

### 5.2 业务记忆模板

```markdown
# 业务 {业务ID}

## 基础信息
- 业务 ID：
- 业务名称：

## 常用执行方案
| 方案名称 | 方案 ID | 用途 | 典型参数 |
|---------|---------|------|---------| 

## 参数偏好
（该业务常用的默认参数值）

## 已知问题
（该业务历史上遇到过的执行问题和解决方案）
```

### 5.3 业务记忆加载流程（强制检查点）

> **🚫 硬性规则：在进入 §6 工作流程之前，必须完成以下业务记忆加载步骤。这不是可选操作。**

**具体执行指令**：

```
步骤 1：确定业务 ID
  → 从用户消息中提取，或询问用户

步骤 2：尝试读取业务记忆文件
  → 使用 read_file 工具读取: {SKILL_DIR}/memory/businesses/{业务ID}.md
  → 文件存在 → 加载内容，记录 memory_biz_loaded = true
  → 文件不存在 → 记录 memory_biz_loaded = false（不影响后续流程）

步骤 3：记忆状态判定
  → memory_biz_loaded = true → 进入「成长期/成熟期」模式，利用业务记忆预填参数
  → memory_biz_loaded = false → 进入「初识期」模式，流程结束后触发记忆沉淀引导（§5.5）
```

**加载后的使用**：
- **业务记忆**：预填常用参数、推荐常用方案
- 无记忆时仅依赖用户输入和方案默认值

### 5.4 引导阶段模型

| 阶段 | 判断条件 | 引导策略 |
|------|---------|---------| 
| **初识期** | 该业务无记忆文件（memory_biz_loaded = false） | 流程结束后引导建立基础业务记忆，每次只问 1-2 个关键信息 |
| **成长期** | 有业务记忆文件但信息不完整 | 在操作过程中自然地补充业务记忆 |
| **成熟期** | 业务记忆信息充分 | 利用业务记忆提高效率（自动预填参数），偶尔确认是否需更新 |

### 5.5 业务记忆沉淀触发点

| 时机 | 引导话术 |
|------|----------| 
| **首次操作某业务**（memory_biz_loaded = false） | 「这是第一次在业务 {ID} 上操作，要不要记录基础信息方便下次使用？」 |
| **用户填写了非默认参数** | 「{参数}={值} 是该业务的常用值吗？要记住吗？」 |
| **重复执行同一方案** | 「该方案已多次被执行，要设为该业务的常用操作吗？」 |

**写入规则**：
1. 展示将要写入的内容 → 用户确认 → 写入文件
2. 若 `memory/` 目录或子目录不存在，自动创建
3. 追加写入，不覆盖已有记忆
4. 写入后将文件加入 git 暂存区

## §6 工作流程

### Step 1：搜索执行方案

1. 确认用户提供了 `bk_scope_type` 和 `bk_scope_id`（若有业务记忆可预填）
2. 调用 `get_job_plan_list`，传入搜索关键词
3. 以表格形式展示结果：

| ID | 名称 | 创建人 | 最后修改时间 |
|----|------|--------|-------------|

4. 如果结果较多，提示用户可以提供更精确的关键词缩小范围
5. 等待用户选择方案或进一步操作

⛔ STOP — 等待用户选择方案

### Step 2：查看方案详情

1. 调用 `get_job_plan_detail` 获取方案详情
2. 分两部分展示：

**【步骤列表】**
| 序号 | 步骤名称 | 类型 |
|------|---------|------|

**【全局变量】**
| 变量名 | 类型 | 默认值 | 来源 |
|--------|------|--------|------|
| {name} | {type} | {default} | [方案默认] |

3. 对未设置默认值的变量（尤其是主机类型），用 ⚠️ 标记并提醒用户需要提供

⛔ STOP — 等待用户确认参数或提供变量值

### Step 3：确认并启动

> **⚠️ 本步骤包含写操作，必须严格遵守 §3.3 确认门禁（GATE CHECK）。**

1. 收集用户提供的变量值，标注每个参数的来源：

| 变量名 | 值 | 来源 |
|--------|-----|------|
| log_path | /data/logs | [方案默认] |
| target_hosts | 127.0.0.1, 127.0.0.2 | [用户提供] |
| env_name | dev | [业务记忆预填] |

2. 展示操作确认摘要（格式见 §3.4）

⛔ STOP — 必须等待用户在**看到上方摘要后**的下一条独立回复作为确认。用户之前表达的任何启动意图（包括触发本次摘要展示的那条消息）都不算确认。

3. **门禁检查**：在调用 `execute_job_plan` 之前，逐项核验 §3.3 的 G1/G2/G3/G4：
   - G1：确认摘要已展示 ✓/✗
   - G2：用户在摘要之后回复了确认 ✓/✗
   - G3：调用参数与摘要一致 ✓/✗
   - G4：全局变量使用 `name` 传递且取自当前方案详情 ✓/✗
   - **全部 ✓ → 调用 `execute_job_plan`**
   - **任一 ✗ → 禁止调用，回到展示摘要步骤**

4. 返回作业实例 ID

### Step 4：查看执行结果

1. 调用 `v4_get_job_instance_status` 查询状态
2. 展示整体状态和各步骤状态：

```
执行状态：{整体状态}
- 步骤 1「{步骤名}」：{状态}
- 步骤 2「{步骤名}」：{状态}
```

3. 提示用户可查看详细日志

### Step 5：查看执行日志（用户需要时）

1. 调用 `v4_batch_get_job_instance_execute_object_log`
2. 按步骤和主机分组展示日志内容

### Step 6：后续引导（含业务记忆沉淀）

1. 提示用户可继续操作（重新搜索、执行其他方案等）
2. **检查业务记忆沉淀触发条件**（§5.5），按以下优先级依次检查：
   - 若 `memory_biz_loaded = false`（初识期）→ 引导记录业务基础信息
   - 若用户本次提供了非默认参数值 → 询问是否记住为该业务的常用参数
   - 若本次方案在业务记忆中已有执行记录 → 询问是否更新
3. 每次最多引导 1 条记忆沉淀，避免打扰用户

## §9 错误处理

| 错误场景 | 处理方式 |
|----------|----------|
| MCP Server 不可用 | 按 §0 引导用户配置 MCP 服务 |
| 认证失败（401/403） | 提示用户检查 access_token 有效性，可能已过期需重新获取 |
| 搜索无结果 | 提示用户更换关键词或确认业务 ID 是否正确 |
| 方案 ID 不存在 | 提示用户确认 ID 是否正确，或重新搜索 |
| 权限不足 | 提示用户联系业务管理员申请权限 |
| 执行失败 | 引导用户查看失败步骤的执行日志，定位失败原因 |
| 参数校验失败 | 检查必填变量是否已填写，主机变量格式是否正确 |
| 网络错误 | 建议检查网络连通性及 VPN 状态 |
